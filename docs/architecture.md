# Architecture

## Overview

Degen Terminal is a serverless, event-driven trading platform built around a single Supabase backend, edge-rendered React frontend, and an external Telegram bot service. The architecture optimizes for three things: **discovery latency** (seeing new launches fast), **execution reliability** (trades land or fail loudly), and **operator economy** (a solo team can run it).

## Design Principles

1. **One backend, many surfaces.** The web terminal, the Telegram bot, and the future mobile app all read from the same Supabase project. State lives in Postgres; clients are thin.
2. **Serverless by default.** Edge functions for ingestion, scoring, and AI calls. No long-running servers except the Telegram bot.
3. **Polling + WebSockets, not webhooks.** External providers' webhooks are unreliable for memecoin discovery. We poll DEX APIs on cron and consume Helius WebSocket events for Solana pool creation.
4. **Custodial UX, custodial-grade security.** Privy provides embedded wallets so users don't need Phantom/MetaMask. Every privileged action requires fresh authentication.
5. **Server-side truth for anything users could spoof.** Trade history, points, referrals — all computed from server-persisted state, never trusted from the client.

## System Layers

### 1. Ingestion

Pulls token, price, and pool data from external sources into Supabase.

**Sources:**
- DexScreener — pair metadata, prices, volumes (BNB + SOL)
- Birdeye — Solana token balances, holders, OHLCV
- Jupiter — Solana token prices, market datas
- Pump.fun, Bags, Four.meme — bonding curve state and graduation signals
- Helius WebSocket — real-time Solana pool creation events
- 1inch — BNB token balances and quotes

**Mechanisms:**
- **Cron-driven edge functions** for tiered polling (hot tokens refreshed more often)
- **Helius WebSocket consumer** running as part of the bot service for sub-second SOL discovery
- **On-demand enrichment** when a user opens a token detail page that hasn't been polled recently

### 2. Storage

A single Supabase Postgres database backs every surface. Core tables include:

- `tokens` — discovered tokens with metadata and current metrics
- `price_snapshots` — time-series price/liquidity data
- `user_trades` — server-of-record for executed trades (powers trade history, rewards, leaderboards)
- `user_alerts` — alert rules with cooldown state
- `user_quests` — quest progress and completion state
- `user_referrals` — referral codes and qualified-referral counts
- `user_roles` — admin/role flags, checked via `has_role()` SQL function
- `telegram_links` — connection state between user accounts and Telegram chat IDs

Row-Level Security (RLS) is enforced on every user-scoped table. The `has_role()` SQL function gates admin endpoints (fees, treasury) at the database level, not just in application code.

### 3. Presentation

A single React + Vite SPA, deployed to Vercel and served from `terminal.degentools.co`.

- **Realtime subscriptions** to Supabase channels for live alpha feed, price updates, and alert state
- **Embedded wallet UX** via Privy — wallet creation, password unlock, and signing all happen in the browser without external extensions
- **Optimistic localStorage cache** for trade history (server is the source of truth; cache is for instant render)
- **Command palette (⌘K)** for global search

### 4. Execution

Trades route through the best aggregator for each chain.

- **Solana:** Jupiter v6 for swaps, Jupiter triggers for limit/conditional orders
- **BNB:** 1inch aggregator
- **Cross-chain:** Li.Fi SDK for BNB ↔ Solana bridging

Every trade is gated behind a fresh safety check (see `safety-system.md`). The Buy/Sell flow surfaces honeypot, mint authority, LP lock, and tax results before the user can confirm.

### 5. Bot Service

A standalone Node.js + grammY service that:
- Listens for `/buy`, `/sell`, `/scan`, `/alert`, etc. commands
- Parses natural-language intent via Gemini 2.5 Flash for non-strict commands
- Executes trades against the same wallets users have on the web terminal
- Runs the Helius WebSocket consumer for live SOL pool events
- Polls active alert rules in a background monitor and dispatches Telegram messages on triggers

The bot links to user accounts via a one-time code generated by the `connect-telegram` edge function.

### 6. AI Layer

Gemini 2.5 Flash, called from edge functions and the bot service:
- **Safety Explain** — converts raw scanner output into plain-English risk summaries
- **Bot intent parsing** — interprets fuzzy commands like "buy 0.1 sol of [contract]"

AI calls are always wrapped with a deterministic fallback. If Gemini is unavailable, the user still gets the structured scanner data — just without the prose summary.

## Why This Stack

| Choice | Why |
|---|---|
| Supabase | One platform for Postgres, edge functions, realtime, auth. Solo-team productivity multiplier. |
| Privy | Embedded wallets without rebuilding key management. Removes the #1 onboarding friction in crypto. |
| Vercel | Edge deploys, preview URLs per branch, zero-config CDN. |
| Jupiter / 1inch / Li.Fi | Best-in-class aggregators per chain. No reason to rebuild routing. |
| Helius WebSocket | Sub-second Solana pool detection beats any polling cadence. |
| grammY | Most ergonomic Telegram framework in TypeScript. |
| Gemini 2.5 Flash | Cheap, fast, good enough for safety summaries and intent parsing. |
| cron-job.org | External cron decouples scheduling from compute. Free and reliable. |

## Trade-offs

- **Custodial wallets.** Privy holds the keys. The trade-off vs. self-custody is real, but the onboarding lift it removes is the difference between users and no users for this audience.
- **Polling cost.** Tiered polling keeps API costs bounded but means cold tokens lag. Acceptable; hot tokens are what matters.
- **Single backend across products.** The Terminal and main DegenTools site share a Supabase project. A backend incident affects both. Worth it at this stage; will split if scale demands.

## Scaling Posture

Every layer scales independently. Edge functions auto-scale. Postgres scales vertically with zero downtime on Supabase. Vercel handles frontend traffic spikes natively. The bot service is the only stateful component; it can be horizontally sharded by user ID if it ever becomes a bottleneck. The realistic ceiling is external API rate limits, addressed via tiered polling and request batching.
